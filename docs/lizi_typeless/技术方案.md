## 当前架构

### Windows 宿主组件

`src/lizi_typeless.Windows` 是 .NET 10 WPF 托盘应用，负责：

1. 使用低级键盘 hook 监听右 Alt，并由 `RightAltStateMachine` 识别双击开始、录音中单击结束和结束后的重复按键抑制。
2. 使用 NAudio WASAPI 从默认多媒体输入设备采集音频，同时写入无头原始 PCM 和可直接播放的 WAV。
3. 显示不激活、不抢焦点的悬浮窗，以及包含播放、复制、Retry 和删除功能的历史窗口。
4. 在开始录音时记录前台窗口句柄和进程 ID；写入前再次验证目标仍是前台窗口。
5. 使用 Unicode `SendInput` 一次性写入最终文本；目标变化或写入失败时保留结果，不自动重试外部副作用。
6. 启动、检查和关闭 WSL2 推理服务，并在流式链路失败时从已保存 WAV 回退到批量转录。

### WSL2 推理组件

`inference/src/lizi_typeless_inference` 是只监听 `127.0.0.1:8765` 的 FastAPI 服务：

1. ASR 使用 `Qwen3-ASR-1.7B`。MVP 后端采用 vLLM 0.14 eager 模式、`max_model_len=4096`、`max_new_tokens=512`、`gpu_memory_utilization=0.50`，并在启动时预热流式路径。
2. 智能整理使用 `Qwen3-1.7B` Transformers bfloat16 推理，固定忠实整理提示词和少量边界示例，并在服务就绪前完成一次生成预热。只有整段由同一个至少 4 字片段连续重复 3 次以上构成时，才在模型前折叠为一次；其他内容不做规则删除。
3. WebSocket 流式接口接收 Windows 原始 PCM 帧；`StreamingAudioConverter` 负责 PCM 解码、声道混合和有状态 SoXR 16 kHz 重采样。
4. ASR 与整理模型共享进程内锁，避免两个 CUDA 路径同时占用同一张 GPU。MVP 只允许一个用户，不建设并发调度层。
5. `scripts/start-inference.sh` 优先选择 `.venv-vllm`；没有该环境时回退到 `.venv` 的 Transformers 批量后端，健康接口会明确返回 `streaming: false`。
6. `vocabulary.py` 只修正当前仍有效且已被个人固定集证明的精确变体：`Cloud Code` 和口述 `/home/litong/logs`；不做模糊匹配。项目改名后删除了只服务旧称呼的纠正规则，也不在没有真实样本时猜测 `lizi_typeless` 的发音变体。扩大 ASR 上下文曾导致 `Kubernetes` 回退为 `Typeless`，因此不采用该方案。

### 本地持久化

每次录音在 `%LOCALAPPDATA%\lizi_typeless\sessions\<session-id>` 下独立保存：

1. `audio.raw`：从 WASAPI 回调持续写入的原始音频，使用 `WriteThrough` 并按短间隔刷盘。
2. `audio.wav`：相同音频的 WAV 容器，供重试和历史播放使用。
3. `session.json`：状态、目标窗口、原始转录、整理结果、失败阶段和分阶段计时。

`SessionStore` 通过同目录临时文件和原子替换更新元数据。应用重启时把遗留的 `recording` 或 `processing` 会话恢复为可 Retry 的 `failed`，并在可能时从 `audio.raw` 重建 WAV。

项目改名时，11 条有效会话从旧数据目录复制到当前目录，并对所有 `session.json`、raw 和 WAV 逐文件执行 SHA-256 对比。旧目录没有删除，当前客户端只读写 `%LOCALAPPDATA%\lizi_typeless`。

## 本地接口

### HTTP

1. `GET /v1/health`：返回模型、设备、就绪和流式能力。
2. `POST /v1/transcribe`：上传完整 WAV，用于 Retry 和流式失败回退。
3. `POST /v1/organize`：接收原始文本并返回忠实整理结果。
4. `POST /v1/shutdown`：只供本机启动该服务的 Windows 客户端在退出时关闭服务。

HTTP JSON 使用 Python `snake_case`。Windows `InferenceClient` 也显式采用 `JsonNamingPolicy.SnakeCaseLower`，并由合同测试覆盖请求和响应字段。

### WebSocket

`/v1/stream` 使用一个连接承载一个录音会话：

1. 客户端先发送包含采样率、声道、位深、编码和 block align 的 `start` 控制消息。
2. 服务返回 `started` 后，客户端按采集顺序发送二进制 PCM 帧。
3. 服务按可用结果返回零到多条 `preview`，只供悬浮窗展示。
4. 客户端发送 `finish`，服务刷出重采样尾帧并返回唯一 `final`。
5. 连接或模型失败时，客户端保留录音并使用完整 WAV 批量转录。

## 主链路与状态

1. 双击右 Alt 后先创建 `recording` 会话，再启动 WASAPI、原始音频落盘和可选流式连接。
2. 结束按键把状态持久化为 `processing`，随后完成流式定稿或批量回退。
3. 原始转录先写入会话，再调用整理服务；最终文本生成后状态变为 `ready`。
4. 只有原窗口仍安全可用时才调用 `SendInput`，成功后变为 `completed`。
5. 任一计算阶段失败都转为 `failed` 并记录可读失败阶段；Retry 从已有原始转录或 WAV 的最近可靠阶段继续，且永不自动插入。

## 计时与诊断

1. 会话记录录音启动、首个音频帧、流式收尾、转录计算、整理计算、文本写入，以及结束按键到各结果阶段的墙钟耗时。
2. Windows 诊断日志为 `%LOCALAPPDATA%\lizi_typeless\diagnostics.log`；推理日志为 `inference/logs/service.log`。
3. 日志记录阶段、耗时和异常；HTTP 错误只保留类型、字段位置和消息，剔除服务回显的请求输入。用户历史文本只存在会话元数据中。
4. `inference/benchmarks/benchmark_service.py` 测量真实 WebSocket 尾延迟；`evaluate_dataset.py` 对固定个人语音集计算 CER、技术词命中率和报告哈希；`summarize_sessions.py` 汇总 Windows 会话的短长录音 P50/P95；`sample_gpu_idle.py` 和 `compare_gpu_idle.py` 生成空闲资源基线报告。

## 构建与运行

从仓库根目录执行：

```bash
./scripts/setup-inference.sh
./scripts/setup-streaming-inference.sh
./scripts/download-models.sh
./scripts/start-inference.sh

dotnet.exe restore lizi_typeless.slnx
dotnet.exe build lizi_typeless.slnx --configuration Debug --no-restore
dotnet.exe test lizi_typeless.slnx --configuration Release --no-restore
```

WPF 不从 WSL UNC 目录直接运行。发布到 Windows 本地目录后启动：

```powershell
dotnet publish src/lizi_typeless.Windows/lizi_typeless.Windows.csproj `
  --configuration Release `
  --output "$env:LOCALAPPDATA\lizi_typeless\app"
& "$env:LOCALAPPDATA\lizi_typeless\app\lizi_typeless.exe"
```

## 安全、隐私与资源边界

1. 服务只绑定回环地址，不暴露到局域网；运行期语音和文本不离开本机。
2. 模型和 Python 环境不进入 Git，个人录音、评测集和报告位于被忽略的 `data/`、`artifacts/` 或 Windows 本地数据目录。
3. 两组各 10 分钟、5 秒间隔的同条件采样显示：模型常驻功耗中位数 74.22 W，应用关闭基线 66.53 W，中位数增量 7.70 W、平均增量 8.93 W；GPU 利用率中位数和 P95 增量均为 0，空闲资源目标通过。
4. vLLM eager 模式冷启动并完成两条预热路径约需 30 秒；客户端在托盘和历史窗口明确展示启动状态。
5. `.venv-vllm` 因上游 `qwen-asr` 把仅用于演示 CLI 的 Gradio 声明为硬依赖，刻意不安装与 vLLM Pydantic 版本冲突的 Gradio；这是当前唯一已知的 `pip check` 缺项，不影响运行路径。

## 当前限制与后续观察

1. 个人真实语音质量基准已完成 20/20 条。首轮人工复核发现数字事实变更和整句否定限制遗漏；当前 `choose_faithful_output` 会在数字序列变化、否定限制减少或内容大段丢失时返回原始转录。第 16、17、20 条定向重放全部通过，修复后完整重放没有新的核心语义回退。
2. 三条 45 至 60 秒长录音均完成，用户确认等待体验良好。整理耗时仍会随输出长度增长，继续作为后续性能优化方向。
3. 浏览器内部焦点、完整应用兼容矩阵和换行细节不阻断个人 MVP，后续按真实使用中出现的 case 优化。
4. 客户端异常退出、推理服务中断和其他低概率故障不做主动破坏性验收；后续在个人真实使用中保留 case、排查并修复。
5. WSL 发行版名称和仓库路径当前由 `InferenceServiceManager` 固定为个人机器实际值，不建设通用安装配置。
