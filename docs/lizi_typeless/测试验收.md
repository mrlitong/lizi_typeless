# lizi_typeless 测试验收

> 状态：已定（个人 MVP 通过）
> 更新日期：2026-07-11（Asia/Shanghai）

本文档是 lizi_typeless 当前范围内验收标准的唯一真相源。本项目是个人本地工具，MVP 采用小样本和真实使用反馈相结合的验收方式，不套用商用产品的大规模矩阵，也不主动制造低概率故障。

## 测量定义

1. **启动响应延迟**：系统确认双击右 Alt 到音频采集已开始且悬浮窗进入录音状态的时间。
2. **结束到写入延迟**：录音状态下按下右 Alt，到最终文本完整出现在目标应用的时间。
3. **短录音**：5 至 15 秒的正常口述。
4. **长录音**：45 至 60 秒、几百字的连续口述。
5. **热状态**：客户端、音频设备和模型均已启动，至少完成一次预热。
6. **原始 CER**：ASR 原始结果相对人工确认原文的中文字符错误率；英文技术词作为完整 token 另行统计。
7. **语义忠实度**：最终文本是否遗漏、增加、反转或改变原始表达的事实、观点、语气和限定条件。

用户未主动反馈等待偏慢时，默认该次等待体验良好，不逐条重复询问；只有用户主动报告慢时才记录为性能问题。

## 当前验证证据

以下结果用于判断当前实现是否可以继续验收，不替代本文件后续要求的样本量：

1. .NET Debug 构建为 0 警告、0 错误；Core 自动化测试 10 项、Windows 合同与目标安全测试 5 项全部通过。
2. Python API、流式音频转换、提示词、忠实度保护、个人词表、重复边界和评测算法测试共 17 项通过，Ruff 检查通过。
3. vLLM 与整理模型可在 RTX 5090 上同时常驻，健康接口返回 `streaming: true`；完成流式与整理预热的冷启动约 30.1 秒，单次观察显存约 22.4 GB。
4. 4.20 秒官方普通话样本按 20 ms 实时音频包发送时，产生 2 次预览，结束到最终转录约 68 ms，整理墙钟约 190 ms，转录与整理文本均正确。
5. 50.45 秒合成长样本按 20 ms 实时发送时，产生 25 次预览，结束到最终转录约 99 ms。规则加入前，154 字重复输入的整理墙钟约 1131 ms；该早期结果定位了长输出性能风险，最终个人体验以第 20 至 22 项的 3 条真实长录音为准。
6. 首条个人实机录音“你好，今天是星期四。”被完整落盘并正确转录；整理阶段因 Windows 请求字段使用 `Text` 而非 `text` 返回 422。根因已修复为统一 `snake_case`，并加入 3 项合同回归测试。
7. 评测工具已对官方短样本完成端到端自测，得到 CER 0 和整理完全匹配；这 1 条官方样本不计入 20 条个人语音验收集。
8. 生产 vLLM 批量回退最初因 64 token 上限截断 50.45 秒 WAV；统一提高到 512 后，完整输出 154 个字符和结尾标点，耗时约 1.48 秒，短样本流式收尾仍约 76 ms。
9. 对整段 11 次完全相同的长片段，窄范围前处理将其折叠为一次，整理计算约 197 ms；两次重复、短促强调和包含其他内容的文本由自动化测试确认不会触发该规则。普通长文本后续已用 3 条真实长录音完成个人体验验收。
10. 用户在修复版中完成旧失败记录 Retry：状态变为 `ready`，整理约 143 ms，未自动写入。随后在 VS Code 安全临时文件完成一条约 6.18 秒真实录音，最终文字准确且只写入一次。
11. 该真实短录音的录音就绪为 98.8 ms，结束到转录 98.0 ms，结束到整理 232.5 ms，结束到写入 246.1 ms，写入阶段 11.0 ms；首帧回调 230.3 ms但未丢首字。
12. 模型常驻与应用关闭各完成 600 秒、5 秒间隔采样。模型常驻功耗中位数 74.22 W、平均 73.10 W；基线中位数 66.53 W、平均 64.17 W；增量分别为 7.70 W 和 8.93 W，低于 10 W。两组 GPU 利用率中位数均为 8%、P95 均为 9%，未发现模型周期性推理抬高利用率。
13. 首批 10 条个人固定集的原始候选为 CER 6.16%、技术词命中 75%；扩大 ASR 上下文虽修复两个词，却误把 `Kubernetes` 识别为 `Typeless`，CER 反升至 6.51%，该方案已撤销。
14. 恢复原上下文并只修正三个当时已观察的精确变体后，同一批音频重放得到 CER 0.68%、原始与最终技术词命中 100%、整理归一化匹配 100%。严格字符串匹配为 60%，其余差异只是标点、空格或合理分段。该结果是前 10 条的阶段基线；后续已扩充到 20 条，其中的旧项目称呼样本不再代表当前名称。
15. 当前 9 条有效短录音结束到写入 P50 466.6 ms、P95 582.3 ms，达到 500/800 ms 目标。录音就绪 P95 110.2 ms，超目标样本来自客户端重启后的首条冷设备录音。该批记录作为个人 MVP 的短录音基线，后续异常随真实使用复查。
16. 应用写入已覆盖 VS Code、Notepad、微信和 Chromium。Chromium 测试发现右 Alt 会影响浏览器内部文本框焦点；用户决定该非核心体验问题和完整应用矩阵移到后续迭代，不阻断个人 MVP。
17. 项目完整改名为 `lizi_typeless` 后，Debug 构建仍为 0 警告、0 错误，Core 10 项、Windows 5 项、Python 14 项测试与 Ruff 全部通过；两套 Python 环境均从新包路径导入 `lizi_typeless_inference`。
18. 11 条有效 Windows 会话已复制到 `%LOCALAPPDATA%\lizi_typeless\sessions`；源目录与目标目录的 33 个会话文件逐一通过 SHA-256 对比，目标目录包含 11 份 raw、11 份 WAV 和 11 份元数据。
19. 新发布的 `%LOCALAPPDATA%\lizi_typeless\app\lizi_typeless.exe` 可正常响应，并从 `/home/litong/lizi_typeless` 启动推理服务；健康接口继续返回两个模型、RTX 5090 和 `streaming: true`。
20. 首条有效长录音为 54.6 秒，结束到转录约 211 ms、结束到整理约 4.66 秒、结束到写入约 4.76 秒，用户确认等待体验良好。除 `lizi 下划线 typeless` 被识别为 `粒子下划线，TypePlus` 外，其余内容完整且技术词正确；该录音已登记为固定集第 11 条。
21. 第二条长录音为 58.9 秒，结束到转录约 155 ms、结束到整理约 3.33 秒，用户认为体验较快。数字、路径、Python 版本、三项列表和明确改口均正确；目标窗口变化使结果保存在历史中而未自动写入，不影响质量和长录音验收计数。
22. 第三条长录音为 55.7 秒，结束到转录约 166 ms、结束到整理约 3.34 秒，用户认为等待较短。问题语气、两个否定条件、本地保存要求、三个失败阶段、目标变化规则和非目标范围全部保留；模型没有回答问题或改变含义。轻微口误仍在最终文本中，但忠实度通过。

24. 第 17 条个人样本的 ASR 两次正确保留“一点三四点二”，整理模型却两次输出 `1.3.2`；目标版本 `1.35.0` 正确。该结果再次证明中文数字到结构化字面量的模型转换不可靠，当时与第 16 条一起列为核心忠实度失败。
25. 第 18 条个人样本的 ASR 与整理结果一致，完整保留百分比和毫秒的中文数字读法，没有改变事实。该样本忠实度通过；未转换为阿拉伯数字只作为格式差异，不阻断 MVP。
26. 第 20 条个人样本的 ASR 原始转录完整包含两项否定限制，整理模型却删除“不要删除历史记录，也不要把语音上传到外部服务”整句。该结果违反关键限定条件不得丢失的核心标准，当时判定为核心忠实度失败。
27. 首轮 20 条完整重放报告为原始 CER 8.55%、最终技术词命中率 75%、整理归一化匹配率 50%。精确指标受旧称呼、数字书写形式和专有名词空格影响，不能单独代表语义质量；人工复核确认第 16、17、20 条为当时的核心忠实度失败。
28. 当前增加数字序列、否定限制和内容保留保护。第 16、17、20 条定向重放全部返回正确原始信息；修复后 20 条完整重放的精确匹配率从 30% 升至 35%、归一化匹配率从 50% 升至 55%，没有新的核心语义回退。技术词格式分数因安全保留中文数字读法而下降，不代表事实准确性回退。

## 验收标准

### 核心交互

1. 空闲时单击右 Alt 不触发录音，双击右 Alt 只创建一个录音会话。
2. 录音时单击或双击右 Alt 都只结束当前会话一次，不立即创建新会话。
3. 实时预览不进入目标输入框；跨应用内部焦点细节不作为个人 MVP 阻断项。
4. 结束处理成功后，最终文本只写入一次。
5. 原目标窗口变化时，不向其他窗口自动写入。
6. 托盘可以查看服务状态、进入历史记录和退出应用。

### 性能

在 RTX 5090、模型已预热且 GPU 没有其他任务的条件下：

| 指标 | 个人 MVP 目标 |
| --- | --- |
| 启动响应延迟 | P95 不超过 100 ms，且测试语音不丢首字 |
| 短录音结束到写入延迟 | P50 不超过 500 ms，P95 不超过 800 ms |
| 长录音结束到写入延迟 | 完成 3 条 45 至 60 秒实录，文本完整、忠实，等待体验可接受；保留各阶段耗时供后续优化 |
| 空闲 GPU 行为 | 连续 10 分钟无周期性推理，GPU 利用率回到空闲水平 |
| 空闲功耗影响 | 相对应用未启动时的同条件基线，稳定增量不超过 10 W |

现有 9 条短录音作为 MVP 短时延基线；长录音测量 3 条。数值阈值用于发现明显回退，不要求以商用产品样本规模证明统计稳定性。长录音即使可接受，也必须保留各阶段耗时供后续优化。

### 转录与智能整理质量

建立至少 20 条个人真实语音的固定测试集，覆盖短句、长段落、技术名词、中英文混输、数字与路径、中途改口、列表和问题语气。

1. 普通话子集的原始 CER 目标不超过 3%。
2. 全部测试集的原始 CER 目标不超过 5%。
3. 预先标注的英文技术名词经允许的个人词表修正后，精确命中率不低于 95%。
4. 智能整理固定用例中不得出现主动回答、事实新增、含义反转或关键限定条件丢失。
5. 用户明确改口时保留最终表达；无法确定是否改口时宁可保留原文，不擅自删除。
6. 只有明确列举或结构清晰的内容才能转换为列表。
7. 每次模型、提示词、词表或推理框架变更后，必须重跑固定测试集并保留结果对比。

CER、技术词和字符串匹配继续作为诊断指标，但个人 MVP 以人工语义忠实度为阻断标准。旧项目称呼、中文数字与阿拉伯数字的书写形式、专有名词空格和已明确后置的个人词典问题不单独阻断；主动回答、事实新增或变更、含义反转和关键限定条件丢失必须为零。

### 数据可靠性与恢复

1. 音频从会话开始持续落盘，不依赖推理服务成功。
2. 真实使用中遇到客户端异常退出时，已经落盘的原始音频应保留；重新启动后应发现未完成会话，并能够恢复或 Retry。
3. 真实使用中遇到推理服务中断时，录音应保留；服务恢复后应能够自动处理或手动 Retry。
4. 程序重启后能发现未完成会话，并将其恢复为可重试或需要确认的状态。
5. ASR 失败后从原始音频重试；整理失败后优先从已保存的原始转录重试。
6. Retry 不删除原始产物，也不自动重复插入已完成会话的文本。
7. 历史数据在没有人工删除时跨应用和系统重启持续存在。

### 应用兼容

以下应用继续作为日常使用目标，不设 MVP 专项次数门槛：

1. 一个 Chromium 浏览器的普通网页输入框和多行编辑框。
2. 微信聊天输入框。
3. VS Code 普通编辑器。
4. Windows Terminal 的 PowerShell 或 shell 提示符。

真实使用中发现焦点、换行、标点或长文本问题时保留对应 case 并针对性优化。密码框、管理员窗口、游戏和全屏应用不作为阻断验收项。

## 自动化测试

当前自动化命令：

```bash
dotnet.exe build lizi_typeless.slnx --configuration Debug --no-restore
dotnet.exe test lizi_typeless.slnx --configuration Release --no-restore
.venv/bin/python -m pytest inference/tests
.venv/bin/ruff check inference
```

自动化覆盖：

1. 右 Alt 双击、录音中结束和重复按键抑制的状态机。
2. 会话原子写入、加载顺序、状态迁移和 Retry 不自动插入。
3. Windows 与 Python 之间的 HTTP JSON 字段合同、错误输入脱敏和禁止向 lizi_typeless 自身窗口自动写入。
4. WebSocket 流式握手、PCM 解码、声道混合和有状态重采样。
5. 智能整理提示词中的禁止回答、扩写和改变原意边界。
6. CER、文本归一化和技术词命中计算。
7. GPU 空闲采样与模型常驻相对基线比较。

推理服务不可用、应用异常退出、流式中断和文本写入失败等低概率场景不主动制造；真实使用出现 case 后保留证据并补针对性覆盖。

固定个人语音集放入被 Git 忽略的 `data/voice-benchmark/`，参考 `inference/benchmarks/manifest.example.jsonl` 建立至少 20 条清单，然后执行：

```bash
.venv-vllm/bin/python inference/benchmarks/evaluate_dataset.py \
  data/voice-benchmark/manifest.jsonl \
  --minimum-samples 20 \
  --output artifacts/voice-benchmark.json
```

真实 WebSocket 尾延迟使用 `inference/benchmarks/benchmark_service.py` 测量；端到端写入延迟以 Windows 会话元数据中的 `endToInsertionMilliseconds` 为准。

长录音过程中可以随时执行以下部分报告。脚本只统计成功自动写入的 `completed` 会话；因非核心焦点问题保存为 `ready` 的长录音，使用会话元数据中的 `endToOrganizationMilliseconds` 和用户体验单独计入 3 条人工验收：

```bash
.venv/bin/python inference/benchmarks/summarize_sessions.py \
  /mnt/c/Users/admin/AppData/Local/lizi_typeless/sessions \
  --allow-partial \
  --output artifacts/session-timings.json
```

空闲资源使用同一采样间隔分别记录模型常驻和应用关闭状态，再生成比较报告：

```bash
.venv/bin/python inference/benchmarks/sample_gpu_idle.py \
  --label models-resident --output artifacts/gpu-idle-models-resident.json
.venv/bin/python inference/benchmarks/sample_gpu_idle.py \
  --label application-stopped --output artifacts/gpu-idle-application-stopped.json
.venv/bin/python inference/benchmarks/compare_gpu_idle.py \
  artifacts/gpu-idle-application-stopped.json \
  artifacts/gpu-idle-models-resident.json \
  --output artifacts/gpu-idle-comparison.json
```

## 人工 QA 场景

1. 日常使用中发现漏触发、重复会话或首字丢失时，保留对应会话并立即记录复现条件。
2. 在自然停顿、连续讲话、快速讲话和中英文切换下观察预览与最终文本。
3. 口述包含“不是 A，是 B”的改口，检查最终文本只在语义明确时删除 A。
4. 口述问题，确认系统只记录问题而不生成答案。
5. 录音时保持原应用焦点，确认悬浮窗不抢焦点；录音中主动切换窗口，确认不会误写。
6. 从历史记录播放音频、查看两个版本文本、复制、Retry 和手动删除。
7. 持续真实使用，把手动纠正、异常退出、服务中断和其他失败样本加入回归集。

## 回归范围

以下变化必须按影响范围重跑固定语音集、相关恢复测试和实际使用应用 smoke test：

1. ASR 模型或推理框架变更。
2. 音频格式、采集参数或分段策略变更。
3. 智能整理模型、提示词或个人词表逻辑变更。
4. Windows 快捷键、焦点跟踪或文本写入逻辑变更。
5. 会话状态、存储格式或重试规则变更。

纯界面文案和不影响行为的布局调整只运行相关 UI 检查。

## 恢复检查

本项目不涉及生产发布回滚。每次引入新的存储格式或模型版本时，必须保证旧历史仍可读取，或者在变更前明确提供一次性本地迁移和备份方式。当前改名没有改变存储格式，迁移后的 11 条会话已经过逐文件哈希校验，旧目录暂时保留用于人工回退。MVP 优先保持格式简单，不提前建设通用迁移框架。

## 已知非阻断区域

1. 个人真实语音质量集 20/20、3 条长录音和忠实度失败修复重放已经完成，个人 MVP 核心验收通过。
2. 异常退出、推理服务中断、Chromium 内部焦点、快捷键压力、完整应用兼容矩阵和风扇主观体验改为真实使用中持续验证和优化。
3. `lizi_typeless` 真实口述、专有名词空格和数字美化继续作为后续个人词典与格式优化方向。
